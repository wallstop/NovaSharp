# Phase A0.5 Memory Cache Retention Research

Phase A0 removed or reduced several hot-path allocations. Phase A0.5 asks the next
question: after those bursts are over, how much memory does NovaSharp keep alive?
The answer matters for Unity scene transitions, editor play-mode churn, compilation
spikes, coroutine churn, and hosted games that load and unload many mods.

The adopted policy is explicit lifecycle control:

- No background trim threads, timers, or finalizers.
- Hosts call `LuaEngine.TrimMemory(...)` at known lifecycle points.
- Shared pools register statistics and trim hooks with `SharedPoolRegistry`.
- Facade statistics combine per-engine script metadata with process-wide NovaSharp
  shared-pool statistics. `ArrayPool<T>.Shared` internals remain opaque.
- Byte estimates are approximate diagnostics, not exact managed object graph sizes.
- Raw native arenas are rejected for v1. Managed region-style scratch owners remain
  allowed only when the lifetime is scoped and correctness-equivalent.

## Adopted API Contract

`LuaEngine.TrimMemory(LuaMemoryTrimLevel level)` is explicit and host-driven. It does
not poll memory pressure and does not start background work. The call trims process-wide
NovaSharp-owned shared pools and the receiving engine's reclaimable compilation-cache entries.

`LuaEngine.GetMemoryStatistics()` returns approximate current retained counts/bytes,
peak retained bytes observed through that engine's statistics facade, trim count, dropped
count, and the current compilation cache entry count. The current retained-byte estimate
combines process-wide NovaSharp shared-pool data with the receiving engine's script-lifetime
metadata, main VM stack, and reachable coroutine stacks. Retention inside `ArrayPool<T>.Shared`
is intentionally excluded because the runtime pool is opaque.

After `TrimMemory(Critical)`, NovaSharp-owned shared pools should retain only configured
minimum entries. Default shared pools retain zero. Live Lua values, rented buffers, active
stack frames, runtime-owned shared-array-pool buckets, and script-lifetime source/bytecode
metadata are not reclaimed by trimming.

## Follow-Up Requirements

- A2 must decide whether packed chunks own reclaimable source/bytecode metadata or whether
  script-lifetime retention remains a documented contract.
- A5 must replace `CallStackItemPool` with struct frames/growable stacks using the same
  registry, statistics, and explicit trim contracts.
- Benchmark-only scratch prototypes compare baseline pool-per-buffer, `ArrayPool<T>`, and
  managed region-style `ScratchScope` for compile scratch and VM scratch. Runtime adoption
  still requires a measured speed or retained-memory win without changing Lua behavior.
